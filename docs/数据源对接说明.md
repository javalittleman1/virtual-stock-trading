---

## 1. 概述

本文档描述模拟炒股应用如何从第三方数据提供商获取 A 股实时行情与历史 K 线数据，并将其同步至 Supabase PostgreSQL 数据库，供前端应用消费。

- **MVP 数据范围**：A 股主板及创业板主要股票（约 2000-3000 只）。
- **更新频率**：交易时段内每 30 秒拉取一次快照；非交易时段暂停拉取以节省配额。
- **数据存储**：Supabase `stocks` 表。

---

## 2. 数据源选型

### 2.1 推荐方案：iTick

| 项目         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **名称**     | iTick（爱 tick）                                             |
| **覆盖市场** | A 股、港股、美股                                             |
| **免费额度** | 注册即享每月 50,000 次免费调用，对于 30 秒/次的 MVP 场景足够 |
| **数据质量** | 实时行情延迟 < 3 秒，提供 WebSocket 和 REST API 两种方式      |
| **接入难度** | 低，提供标准 HTTP API，返回 JSON 格式                        |
| **官网**     | https://itick.org                                            |

### 2.2 备选方案

| 方案            | 适用场景                               | 备注                             |
| --------------- | -------------------------------------- | -------------------------------- |
| **Yahoo Finance** | 美股为主，A 股数据延迟较大（15 分钟）  | 免费无认证，适合开发测试         |
| **新浪财经**      | A 股数据较全，但无官方 API，稳定性差   | 仅作兜底，不推荐生产使用         |
| **TuShare**       | 适合 Python 后端，提供丰富历史数据     | 若后端使用 Python 可考虑         |

> **MVP 阶段决策**：采用 iTick REST API，未来可升级为 WebSocket 以降低延迟。

## 3. API 调用规范

### 3.1 认证方式

所有请求需在 Header 中携带 API Key：

```
Authorization: Bearer <ITICK_API_KEY>
```

### 3.2 获取单只股票实时行情

**GET** `https://api.itick.org/v1/quote/{symbol}`

**示例请求**：
```
GET https://api.itick.org/v1/quote/600519
Authorization: Bearer YOUR_API_KEY
```

**响应示例**：
```json
{
  "symbol": "600519",
  "name": "贵州茅台",
  "price": 1685.00,
  "prev_close": 1683.18,
  "open": 1684.00,
  "high": 1692.50,
  "low": 1678.00,
  "volume": 1234567,
  "timestamp": "2026-04-09T10:30:00Z"
}
```

### 3.3 批量获取多只股票实时行情（推荐）

**GET** `https://api.itick.org/v1/quotes`

**查询参数**：

| 参数    | 类型   | 说明                                         |
| ------- | ------ | -------------------------------------------- |
| symbols | string | 股票代码列表，逗号分隔，最多 50 只           |
| fields  | string | 返回字段，默认全部                           |

**示例请求**：
```
GET https://api.itick.org/v1/quotes?symbols=600519,000858,300750
Authorization: Bearer YOUR_API_KEY
```

**响应示例**：
```json
{
  "data": [
    { "symbol": "600519", "price": 1685.00, "volume": 1234567, ... },
    { "symbol": "000858", "price": 142.30, "volume": 987654, ... },
    ...
  ]
}
```

> **MVP 策略**：使用批量接口，每次拉取全部监控股票列表（约 2000-3000 只），分 40-60 个批次并行请求，控制在 30 秒内完成一轮全量更新。

### 3.4 获取历史 K 线数据

**GET** `https://api.itick.org/v1/kline/{symbol}`

**查询参数**：

| 参数   | 类型   | 说明                                  |
| ------ | ------ | ------------------------------------- |
| period | string | `day`、`week`、`month`                |
| limit  | number | 返回条数，默认 100，最大 500          |

**示例请求**：
```
GET https://api.itick.org/v1/kline/600519?period=day&limit=100
Authorization: Bearer YOUR_API_KEY
```

**响应示例**：
```json
{
  "data": [
    { "date": "2026-04-08", "open": 1680.00, "high": 1690.00, "low": 1675.00, "close": 1683.18, "volume": 2345678 },
    ...
  ]
}
```

> **使用场景**：K 线图组件按需加载历史数据，前端直接调用 Next.js API 路由（该路由代理至 iTick），不写入 Supabase。

## 4. 数据同步流程

### 4.1 整体架构

```
Vercel Cron Job (每30秒触发)
        │
        ▼
/api/cron/update-prices (Next.js API Route)
        │
        ├─ 1. 从 Supabase 获取需监控的股票列表
        ├─ 2. 分批调用 iTick 批量行情接口
        ├─ 3. 合并数据，批量 UPSERT 到 Supabase stocks 表
        └─ 4. Supabase Realtime 推送变更给前端
```

### 4.2 定时任务实现（Vercel Cron Job）

在 `vercel.json` 中配置：

```json
{
  "crons": [
    {
      "path": "/api/cron/update-prices",
      "schedule": "*/30 * * * *"
    }
  ]
}
```

> **注意**：Vercel 免费版 Cron Job 仅支持每日执行一次，生产环境需升级至 Pro 版，或使用外部服务（如 EasyCron、GitHub Actions）触发。**MVP 阶段可暂用手动触发或简化频率**。

### 4.3 API 路由核心逻辑

```typescript
// app/api/cron/update-prices/route.ts
import { NextResponse } from 'next/server';
import { supabase } from '@/lib/supabase';

const ITICK_API_KEY = process.env.ITICK_API_KEY!;
const BATCH_SIZE = 50;

export async function GET() {
  // 1. 判断是否在 A 股交易时段（9:30-11:30, 13:00-15:00），非交易时段跳过
  if (!isTradingHour()) {
    return NextResponse.json({ message: '非交易时段，跳过更新' });
  }

  // 2. 获取所有股票代码
  const { data: stocks } = await supabase.from('stocks').select('symbol');
  const symbols = stocks?.map(s => s.symbol) || [];

  // 3. 分批获取行情
  for (let i = 0; i < symbols.length; i += BATCH_SIZE) {
    const batch = symbols.slice(i, i + BATCH_SIZE);
    const res = await fetch(
      `https://api.itick.org/v1/quotes?symbols=${batch.join(',')}`,
      { headers: { Authorization: `Bearer ${ITICK_API_KEY}` } }
    );
    const { data } = await res.json();
    
    // 4. 批量更新 Supabase
    const updates = data.map((item: any) => ({
      symbol: item.symbol,
      current_price: item.price,
      prev_close: item.prev_close,
      open: item.open,
      high: item.high,
      low: item.low,
      volume: item.volume,
      updated_at: new Date().toISOString(),
    }));
    
    await supabase.from('stocks').upsert(updates, { onConflict: 'symbol' });
  }

  return NextResponse.json({ success: true, updated: symbols.length });
}

function isTradingHour(): boolean {
  const now = new Date();
  const hour = now.getHours();
  const minute = now.getMinutes();
  const time = hour * 100 + minute;
  // 周一至周五
  if (now.getDay() === 0 || now.getDay() === 6) return false;
  // 9:30-11:30 或 13:00-15:00
  return (time >= 930 && time <= 1130) || (time >= 1300 && time <= 1500);
}
```

### 4.4 初始数据填充

部署后需运行一次性脚本，将 A 股股票基础信息（代码、名称、市场）导入 `stocks` 表。可从 iTick 获取全量股票列表接口填充。

## 5. 降级与容错策略

| 场景                       | 处理方式                                                     |
| -------------------------- | ------------------------------------------------------------ |
| iTick API 超时或返回 5xx   | 记录错误日志，跳过本次更新，保留上一轮数据；连续失败 3 次后发送告警 |
| 单只股票拉取失败           | 不影响其他股票，该股票价格暂时不更新                          |
| Supabase 写入失败          | 重试 2 次，仍失败则记录日志                                   |
| Vercel Cron Job 未触发     | 前端可提供「手动刷新」按钮，调用同一接口                      |
| API 免费额度即将耗尽       | 监控调用次数，接近上限时降低更新频率或切换至备选数据源        |

## 7. MVP 阶段简化方案

为快速验证产品，MVP 阶段可采用以下简化策略：

| 项目               | MVP 简化方案                                                 | 后续优化                           |
| ------------------ | ------------------------------------------------------------ | ---------------------------------- |
| **定时触发**       | 使用 Vercel Cron Job（Pro 版）或 GitHub Actions 每 5 分钟触发一次 | 升级为专业 Cron 服务，提高频率     |
| **股票数量**       | 仅监控沪深 300 成分股 + 用户自选股（动态增减）               | 扩展至全市场                       |
| **数据更新**       | 交易时段内每 5 分钟全量更新一次                              | 30 秒增量更新                      |
| **K 线数据**       | 前端直接调用 iTick API（不缓存）                             | 增加 Redis 缓存层                  |
| **错误处理**       | 简单的 try-catch + console.error                             | 接入 Sentry 等错误监控             |

---

## 8. 数据源切换预案

若 iTick 服务不可用或费用过高，可快速切换至备选方案：

| 切换步骤                     | 说明                                         |
| ---------------------------- | -------------------------------------------- |
| 1. 修改 API 路由中的请求地址 | 将 iTick 域名替换为 Yahoo Finance 或其他源   |
| 2. 调整响应字段映射           | 不同数据源字段名可能不同，需做适配           |
| 3. 更新环境变量               | 替换 API Key                                 |
| 4. 重新部署                   | Vercel 自动部署                              |

建议在代码中封装一个 `DataProvider` 接口层，便于日后无缝切换。
